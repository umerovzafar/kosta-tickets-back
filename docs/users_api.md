# Инструкция по запросам Users API

Все запросы выполняются через **Gateway**. Базовый URL: `http://localhost:1234` (или ваш адрес Gateway).

Во всех запросах (кроме получения своего профиля) требуется авторизация. Заголовок:
```
Authorization: Bearer <access_token>
```

---

## 1. GET /api/v1/users/me — Get Me

**Назначение:** получение профиля текущего авторизованного пользователя.

**Метод:** `GET`  
**Путь:** `/api/v1/users/me`

**Права доступа:** любой авторизованный пользователь (нужен валидный токен).

**Заголовки:**
| Заголовок           | Значение              |
|---------------------|------------------------|
| `Authorization`      | `Bearer <access_token>` |

**Параметры:** нет.

**Пример cURL:**
```bash
curl -X GET "http://localhost:1234/api/v1/users/me" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

**Успешный ответ (200):**
```json
{
  "id": 1,
  "email": "user@example.com",
  "display_name": "Иван Иванов",
  "picture": "https://...",
  "role": "Сотрудник",
  "position": "Юрист",
  "is_blocked": false,
  "is_archived": false,
  "created_at": "2024-01-15T10:00:00",
  "updated_at": "2024-01-20T12:00:00",
  "time_tracking_role": "user"
}
```

| Поле                | Тип     | Описание |
|---------------------|---------|----------|
| `id`                | int     | Идентификатор пользователя. |
| `email`             | string  | Email. |
| `display_name`      | string? | Отображаемое имя. |
| `picture`           | string? | URL аватара. |
| `role`              | string  | Роль в системе (например: Сотрудник, Администратор, Главный администратор, Партнер, IT отдел, Офис менеджер). |
| `position`          | string? | Должность. |
| `is_blocked`        | bool    | Заблокирован ли пользователь. |
| `is_archived`       | bool    | В архиве ли пользователь. |
| `created_at`        | string  | Дата создания (ISO 8601). |
| `updated_at`        | string? | Дата обновления. |
| `time_tracking_role`| string? | Роль в учёте времени: `"user"`, `"manager"` или `null`. |

**Ошибки:** 401 — не передан или неверный/истёкший токен.

## 2. GET /api/v1/users — List Users

**Назначение:** получение списка пользователей системы.

**Метод:** `GET`  
**Путь:** `/api/v1/users`

**Права доступа:** любой авторизованный пользователь (нужен валидный токен).

**Заголовки:**
| Заголовок      | Значение              |
|----------------|------------------------|
| `Authorization`| `Bearer <access_token>` |

**Query-параметры:**
| Параметр           | Тип   | По умолчанию | Описание |
|--------------------|-------|--------------|----------|
| `include_archived` | bool  | `false`      | Включить в список архивированных пользователей. |

**Пример cURL:**
```bash
curl -X GET "http://localhost:1234/api/v1/users?include_archived=false" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

**Успешный ответ (200):** массив объектов в формате профиля (как в Get Me), например:
```json
[
  {
    "id": 1,
    "email": "user1@example.com",
    "display_name": "Иван",
    "picture": null,
    "role": "Сотрудник",
    "position": "Юрист",
    "is_blocked": false,
    "is_archived": false,
    "created_at": "2024-01-15T10:00:00",
    "updated_at": null,
    "time_tracking_role": null
  }
]
```

**Ошибки:** 401 — не авторизован.

## 3. GET /api/v1/users/{user_id} — Get User Detail

**Назначение:** получение подробной информации о конкретном пользователе по его ID.

**Метод:** `GET`  
**Путь:** `/api/v1/users/{user_id}`

**Права доступа:** любой авторизованный пользователь (нужен валидный токен).

**Заголовки:**
| Заголовок      | Значение              |
|----------------|------------------------|
| `Authorization`| `Bearer <access_token>` |

**Параметры пути:**
| Параметр  | Тип | Описание |
|-----------|-----|----------|
| `user_id` | int | Идентификатор пользователя. |

**Пример cURL:**
```bash
curl -X GET "http://localhost:1234/api/v1/users/5" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

**Успешный ответ (200):**
```json
{
  "id": 5,
  "azure_oid": "abc-123-...",
  "email": "user@example.com",
  "display_name": "Иван Иванов",
  "picture": null,
  "role": "Сотрудник",
  "position": "Юрист",
  "is_blocked": false,
  "is_archived": false,
  "time_tracking_role": "user",
  "created_at": "2024-01-15T10:00:00",
  "updated_at": "2024-01-20T12:00:00"
}
```

Дополнительно к полям из Get Me приходит поле `azure_oid` (идентификатор в Microsoft Azure).

**Ошибки:**
- 401 — не авторизован.
- 404 — пользователь с таким `user_id` не найден.

## 4. PATCH /api/v1/users/{user_id}/role — Set User Role

**Назначение:** назначение роли пользователю в системе (Сотрудник, Администратор, Главный администратор и т.д.).

**Метод:** `PATCH`  
**Путь:** `/api/v1/users/{user_id}/role`

**Права доступа:** только **Главный администратор**.

**Заголовки:**
| Заголовок       | Значение               |
|-----------------|------------------------|
| `Authorization` | `Bearer <access_token>` |
| `Content-Type`   | `application/json`      |

**Параметры пути:**
| Параметр  | Тип | Описание |
|-----------|-----|----------|
| `user_id` | int | Идентификатор пользователя. |

**Тело запроса (JSON):**
| Поле  | Тип    | Обязательное | Описание |
|-------|--------|--------------|----------|
| `role`| string | да           | Название роли (должно совпадать с одной из ролей в системе). |

**Пример тела:**
```json
{
  "role": "Администратор"
}
```

Допустимые значения: `Сотрудник`, `IT отдел`, `Партнер`, `Администратор`, `Главный администратор`, `Офис менеджер`.

**Пример cURL:**
```bash
curl -X PATCH "http://localhost:1234/api/v1/users/5/role" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"role\": \"Администратор\"}"
```

**Успешный ответ (200):** объект пользователя в формате Get User Detail (с обновлённым полем `role`).

**Ошибки:**
- 401 — не авторизован.
- 403 — только Главный администратор может назначать роли.
- 404 — пользователь не найден или роль не существует.

## 7. PATCH /api/v1/users/{user_id}/time-tracking-role — Set Time Tracking Role

**Назначение:** назначить пользователю роль в модуле учёта времени: `user` (ведение учёта) или `manager` (управление). Либо сбросить роль (`null`).

**Метод:** `PATCH`  
**Путь:** `/api/v1/users/{user_id}/time-tracking-role`

**Права доступа:** **Главный администратор**, **Администратор** или **Партнер**.

**Заголовки:**
| Заголовок       | Значение               |
|-----------------|------------------------|
| `Authorization` | `Bearer <access_token>` |
| `Content-Type`  | `application/json`      |

**Параметры пути:**
| Параметр  | Тип | Описание |
|-----------|-----|----------|
| `user_id` | int | Идентификатор пользователя. |

**Тело запроса (JSON):**
| Поле                | Тип     | Обязательное | Описание |
|---------------------|---------|--------------|----------|
| `time_tracking_role`| string? | нет          | `"user"`, `"manager"` или `null` (сброс роли). |

**Примеры тела:**
```json
{ "time_tracking_role": "user" }
```
```json
{ "time_tracking_role": "manager" }
```
```json
{ "time_tracking_role": null }
```

**Пример cURL:**
```bash
curl -X PATCH "http://localhost:1234/api/v1/users/5/time-tracking-role" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"time_tracking_role\": \"manager\"}"
```

**Успешный ответ (200):** объект пользователя в формате Get User Detail (с обновлённым полем `time_tracking_role`).

**Ошибки:**
- 400 — недопустимое значение `time_tracking_role` (допустимы только `user`, `manager` или `null`).
- 401 — не авторизован.
- 403 — недостаточно прав.
- 404 — пользователь не найден.
