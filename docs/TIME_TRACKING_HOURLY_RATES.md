# Учёт времени: почасовые ставки пользователей

Инструкция по работе с **оплачиваемыми** и **себестоимостными** ставками по умолчанию для сотрудника. Все запросы идут через **gateway** с префиксом `/api/v1/time-tracking`.

## Таблицы в PostgreSQL

Сервис `time_tracking` при старте создаёт таблицы через SQLAlchemy (`create_all`). Для ручного развёртывания см. скрипт в репозитории: `scripts/create_time_tracking_db_tables.sql`. Удаление: `scripts/drop_time_tracking_db_tables.sql`.

## Предварительные условия

1. **Авторизация** — заголовок `Authorization: Bearer <access_token>` (как для остального API).
2. Пользователь должен быть в списке учёта времени: запись в БД `time_tracking` синхронизируется с auth (тот же `auth_user_id`, что в системе пользователей). **Список** ставок (`GET .../hourly-rates`) для пользователя, которого ещё нет в учёте времени, возвращает **пустой массив `[]`**, а не 404. Для **создания** ставки пользователь уже должен быть в учёте времени; иначе — 404 (см. `GET/POST /api/v1/time-tracking/users`).

## Два типа ставок

| Тип (`rate_kind`) | Назначение |
|-------------------|------------|
| `billable` | Оплачиваемая ставка — по какой ставке клиент оплачивает время сотрудника (для проектов с «ставкой по умолчанию»). |
| `cost` | Ставка себестоимости — внутренние затраты на сотрудника, на все проекты. |

Интервалы дат **внутри одного типа** у одного пользователя **не должны пересекаться**. Пустое **начало** = «с начала времён», пустое **окончание** = «без ограничений».

## Права (роли из auth)

| Действие | `billable` | `cost` |
|----------|------------|--------|
| Просмотр списка / одной ставки | Главный администратор, Администратор, Партнёр, IT отдел, Офис менеджер | Только Главный администратор и Администратор |
| Создание, изменение, удаление | Главный администратор, Администратор, Партнёр | Только Главный администратор и Администратор |

Сотрудник без ролей из таблицы **не** получает доступ к этим эндпоинтам.

## Базовый URL

```
{GATEWAY}/api/v1/time-tracking
```

Пример: `https://ticketsback.kostalegal.com/api/v1/time-tracking`.

### Альтернативный путь (тот же API)

Чтобы обойти **404 от nginx**, если прокси настроен только на `/api/v1/users`, те же операции доступны под:

```
{GATEWAY}/api/v1/users/{auth_user_id}/hourly-rates
{GATEWAY}/api/v1/users/{auth_user_id}/hourly-rates/{rate_id}
```

Пример: `GET https://ticketsback.kostalegal.com/api/v1/users/1/hourly-rates?kind=billable` — эквивалент `GET .../time-tracking/users/1/hourly-rates?kind=billable`.

### Если всё ещё 404 на проде

1. Пересоберите и задеплойте **gateway** с актуальным кодом.
2. В **nginx** должен проксироваться весь префикс API, например `location /api/` → upstream gateway, а не отдельные ветки только под `users` или `expenses`.

## API

### Список ставок пользователя

```http
GET /users/{auth_user_id}/hourly-rates?kind=billable
GET /users/{auth_user_id}/hourly-rates?kind=cost
```

Обязательный query-параметр **`kind`**: `billable` или `cost`.

**Ответ** — массив объектов (поля в JSON — **snake_case**):

| Поле | Описание |
|------|----------|
| `id` | UUID строки ставки |
| `auth_user_id` | ID пользователя в auth |
| `rate_kind` | `billable` или `cost` |
| `amount` | Сумма за час (decimal) |
| `currency` | Код валюты, напр. `USD` |
| `valid_from` | Дата начала или `null` |
| `valid_to` | Дата окончания или `null` |
| `created_at`, `updated_at` | UTC |

---

### Одна ставка по id

```http
GET /users/{auth_user_id}/hourly-rates/{rate_id}
```

Права те же, что и на просмотр соответствующего типа (`billable` / `cost`).

---

### Создание ставки

```http
POST /users/{auth_user_id}/hourly-rates
Content-Type: application/json
```

Тело (удобно для фронта — **camelCase**; gateway принимает и преобразует):

```json
{
  "rateKind": "billable",
  "amount": "100.00",
  "currency": "USD",
  "validFrom": null,
  "validTo": null
}
```

- **`rateKind`**: обязательно `billable` или `cost`.
- **`amount`**: больше нуля.
- **`currency`**: по умолчанию можно не передавать — `USD`.
- **`validFrom` / `validTo`**: дата в формате ISO `YYYY-MM-DD` или `null` (без ограничения с начала / без конца).

При пересечении с другой ставкой того же типа — **400** с текстом вроде «Интервал пересекается с другой ставкой этого типа». Если начало позже конца — **400** («Дата начала не может быть позже даты окончания»).

---

### Изменение ставки

```http
PATCH /users/{auth_user_id}/hourly-rates/{rate_id}
Content-Type: application/json
```

Передавайте только меняемые поля (snake_case или camelCase согласно вашему клиенту; gateway для PATCH сериализует в **snake_case**):

```json
{
  "amount": "120.00",
  "currency": "USD",
  "validFrom": "2025-01-01",
  "validTo": null
}
```

Чтобы **сбросить** границу на «без ограничения», передайте для поля значение `null` явно (если поле передано в теле).

Пустое тело без полей — **400** («Нет полей для обновления»).

---

### Удаление ставки

```http
DELETE /users/{auth_user_id}/hourly-rates/{rate_id}
```

Ответ при успехе: `{"ok": true}`.

---

## Пользователи учёта времени (справочно)

Список тех, кому вообще настраиваются ставки (кто есть в разделе учёта времени):

```http
GET /api/v1/time-tracking/users
```

Требуются права просмотра учёта времени (см. gateway). Идентификатор **`id`** в ответе — это **`auth_user_id`** для путей `/users/{auth_user_id}/hourly-rates/...`.

---

## Фронтенд

- Базовый путь к API: `/api/v1/time-tracking/...` относительно origin gateway (локально через Vite proxy — см. `docs/ENV_AND_FRONTEND.md`).
- Использовать общий `apiFetch` с Bearer-токеном.
- Для форм «новая ставка» соответствовать полям `rateKind`, `validFrom`, `validTo`, `amount`, `currency` как выше.

## Ошибки

| Код | Когда |
|-----|--------|
| 401 | Нет или невалидный токен |
| 403 | Недостаточно прав для типа ставки (billable / cost) |
| 404 | Пользователь не в учёте времени или ставка не найдена |
| 400 | Валидация (пересечение интервалов, сумма ≤ 0, даты, пустой PATCH) |
| 503 | Сервис `time_tracking` недоступен или не настроен на gateway |
